# 결정 기록

마지막 갱신일: 2026-05-26

## 2026-05-26 — 초기 스택

**결정:** 웹 서버는 Next.js 15 (App Router, TypeScript). LLM은 Gemini 2.5 Flash. 배포는 Vercel 무료 티어 가정.

**이유:**
- `gijilai`가 같은 스택이라 익숙하고, 도구/배포 경험을 재사용한다.
- App Router 하나로 프론트 + API endpoint를 같이 두면 thin-slice 사이클이 가장 짧다.
- Gemini 2.5 Flash는 한국어 품질/응답 속도가 좋고, 무료 티어로 검증 단계를 굴릴 수 있다.

**범위:** Explore 단계 thin-slice. 실제 사주 풀이 정밀도보다 "문제 입력 → 컨설팅 → 재방문" 루프 검증이 목적.

## 2026-05-26 — Gemini 무료 티어 사용

**결정:** Google AI Studio 무료 티어 키로 시작.

**이유:** Explore 단계라 비용 0이 핵심. 베타 사용자 수가 적어 rate limit 안에 들어옴.

**리스크:**
- **무료 티어 입력은 모델 학습에 사용될 수 있음** (Google AI Studio 약관). 사용자 생년월일시·개인 고민이 학습 데이터로 흘러갈 가능성을 사용자에게 고지해야 함.
- Rate limit (분당 요청 수 제한) — 동시 접속이 늘면 막힘.
- SLA 없음.

**전환 트리거 (paid tier로 가야 하는 시점):**
- 실제 결제 사용자가 발생할 때.
- 베타라도 외부 사용자에게 공개해 트래픽이 분당 한도에 닿을 때.
- 입력 데이터 보호를 약속해야 할 때 (개인정보 처리방침 명시).

**전환 시 작업:** API 키를 paid tier로 교체, 개인정보 처리방침과 약관에 LLM 처리 범위 명시, 학습 옵트아웃 확인.

## 2026-05-26 — 사주 계산은 일단 LLM 위임

**결정:** 만세력/천간지지 정밀 계산 라이브러리 도입을 보류하고, 생년월일시를 그대로 Gemini 프롬프트에 넘겨 LLM이 풀이까지 한다.

**이유:** thin-slice 목적은 "컨설팅 루프"가 도는지 검증이지, 풀이 정확도 자체가 아니다. 정확도 부족이 사용자 이탈 주된 이유로 나오면 그때 만세력 엔진을 붙인다.

**다음 결정 신호:** 베타 사용자 5명 이상이 "풀이가 부정확해서 못 믿겠다"고 지적하면 만세력 엔진 도입 결정.

## 2026-05-26 — (번복) 만세력 라이브러리 즉시 도입

**결정:** 이전 결정("사주 계산 LLM 위임")을 같은 날 번복. `lunar-javascript`로 천간·지지·일주·납음·오행 분포·띠를 결정론적으로 계산하고, LLM에는 계산된 값을 그대로 인용해 해석만 시킨다.

**번복 이유:**
- 이 프로토타입의 목적은 "프롬프트 튜닝"인데, LLM이 사주를 계산하면 같은 입력에도 천간지지가 들쭉날쭉 나옴. 그러면 결과 변화의 원인을 "프롬프트 차이"인지 "계산 노이즈"인지 분리할 수 없음.
- 만세력은 정해진 규칙(절기 기반 월주 등)이라 라이브러리 사용이 자연스럽고, 계산은 deterministic해야 마땅함.

**구현:**
- `lib/saju/calculator.ts` — `lunar-javascript`(MIT) 래퍼. 사주 4기둥 + 일간 + 오행 분포 + 띠 반환.
- `lib/saju/readings.ts` — 천간·지지·오행·띠의 한자→한글 매핑.
- `lib/saju/format.ts` — 프롬프트 주입용 문자열 포매터.
- 3개 리포트 라우트(`/api/saju/personal`, `/api/family/report`, `/api/tci/report`)가 호출 시점에 계산해서 `{{sajuTable}}`, `{{dayMaster}}` 등으로 주입.
- 프롬프트는 "주어진 사주만 사용하고 다시 계산하지 말 것"을 명시.

**한계:**
- 윤달, 분 단위 시주 경계, 서머타임/표준시 보정은 prototype 수준. 정밀도가 요구되면 보정 로직 추가 필요.
- 십신(十神) 자체는 라이브러리에서 뽑을 수 있으나 현재 프롬프트에 미포함. 필요해지면 `formatSajuForPrompt`에 추가.

## 2026-05-26 — 프로토타입 목적과 구조

**결정:** sajulife 첫 버전의 목적은 "AI 사주 리포트 프롬프트 튜닝 환경"이다. 정식 서비스가 아니라 프롬프트 반복 실험용 프로토타입.

**구조:**
- 게스트 로그인 (쿠키 기반, 회원가입 없음)
- 사주 정보 입력 → 대시보드 → 3개 시나리오(기질 / 개인 사주 / 가족 사주)
- 각 시나리오마다 AI 리포트 + 프롬프트 디버그 화면을 한 쌍으로 둔다.
- 디버그 화면에서 수정한 프롬프트는 파일에 영속 (`data/prompts.json`).
- 게스트별 데이터(사주, 설문 답안, 가족)는 `data/guests/{guestId}/*.json`.

**이유:** 회원/DB 없이 빠르게 프롬프트 결과를 비교/조정하려는 목적. 운영 부담을 최소화하고 dev 환경 단일 프로세스 가정.

## 2026-05-26 — AI 공급자 추상화

**결정:** `lib/ai/types.ts`의 `AIProvider` 인터페이스 + `lib/ai/index.ts`의 `getAIProvider()` 팩토리. 환경변수 `AI_PROVIDER`(기본 `gemini`)로 교체.

**이유:** Gemini 무료 티어로 시작하지만, paid 전환 또는 다른 모델(Claude, OpenAI 등)로 갈아탈 가능성 높음. 각 호출부는 인터페이스만 알면 되도록 격리.

## 2026-05-26 — 기질 검사는 자체 프로토타입 문항

**결정:** 정식 TCI(Cloninger) 문항은 저작권으로 임의 사용 불가. 프로토타입에서는 7차원 구조(자극추구·위험회피·사회적민감성·인내력·자율성·연대감·자기초월)만 차용하고 문항은 직접 작성한 자체 문항 35개(차원당 5개)로 진행.

**이유:** thin-slice 목적이 "리포트 프롬프트 튜닝"이지 정밀 측정이 아니므로 자체 문항으로 충분.

**전환 트리거:** 측정 정확도가 실제로 필요해지면 정식 TCI 라이선스 또는 다른 검증된 도구(Big Five, NEO-PI 등) 도입을 검토.
